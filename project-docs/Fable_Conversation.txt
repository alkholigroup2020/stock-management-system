I've read the plan and cross-checked it against the actual codebase (97 server files import the singleton Prisma client, including shared utils like documentNumbering.ts and stockValidation.ts). Here's my assessment.

Verdict: the concept is sound and genuinely doable — but the plan has one factual flaw, one architectural mismatch, and a timeline that's optimistic for the real scope

The database-per-tenant + control-plane pattern is a legitimate, well-trodden SaaS architecture, and it's actually a particularly good fit for this app: because tenant isolation lives at the database level, the existing schema needs zero changes — no organization_id retrofit, no touching production data. Your current AKG deployment can become "tenant #1" by simply registering its existing database in the control plane. That's the strongest argument for this plan, and the plan is right to choose it over row-level isolation.

That said, three things need correcting before you build it:

1. The Supabase strategy contradicts itself (factual problem)

The Tenant model stores database_name and buildConnectionUrl() connects to it — that's database-per-tenant. But Phase 8's cost table says Starter/Pro tenants use a "shared project, separate schemas". Supabase doesn't support multiple databases in one project — each project is designed around a single postgres database. So "shared project" necessarily means schema-per-tenant, and the plan's data model has no database_schema field and its connection URL builder can't express it. You'd need to:

- Add a database_schema column and append ?schema=<tenant> to the Prisma connection string, and
- Verify with a spike that Prisma's schema parameter behaves correctly through Supabase's transaction pooler (port 6543, pgbouncer=true) — search_path handling through transaction-mode pooling has historically been a sharp edge. This is a day-one spike, not a week-7 discovery.

The dedicated-project-per-tenant path (Enterprise) is fine as written, and Supabase's Management API can automate provisioning.

2. The connection manager assumes a long-running server; you deploy to Vercel serverless

The TenantConnectionManager (LRU of 50 PrismaClients, idle cleanup timers, periodic cleanupIdle()) is designed for a persistent Node process. On Vercel, each function instance gets its own copy of that cache, instances come and go, and nothing calls your cleanup loop. It won't break — Fluid Compute reuses instances so caching still helps — but the design needs reframing: per-instance memoization with connection_limit=1 per client, a TTL cache for control-plane tenant lookups (as written, every API request pays a control-plane query before doing any work), and acceptance that eviction/$disconnect() is best-effort. Also, the LRU eviction as written can disconnect a client mid-query in a concurrent request. All fixable, but the Phase 5 code shouldn't be built as-is.

3. The "settings" section quietly triples the scope

Feature flags (hide the Transfers menu, gate PDF export) are cheap — the plan's estimate is fair. But TenantSettings includes allowNegativeStock, requirePOForDelivery, autoGeneratePriceVarianceNCR, and configurable VAT. Those aren't settings — they're forks in your core business invariants, which the codebase currently enforces as hard rules in many places (no-negative-stock validation, mandatory PO link, zero-tolerance NCR). Each one is a business-logic branch with its own test burden. I'd cut these from v1 entirely and ship with today's rules for all tenants. Similarly, enableBarcodeScanning flags a feature that doesn't exist yet.

Smaller but real gaps

- The 103-route migration is bigger than find-and-replace. Besides the 90+ API routes, shared utils (documentNumbering, stockValidation, ncrCredits) import the singleton client at module scope and must become parameterized, and the async email pipeline needs tenant context threaded through it.
- The dev-only ?tenant= override must be hard-gated on environment — in production it would be a tenant-hopping vulnerability.
- Wildcard subdomains on Vercel require moving the domain to Vercel nameservers — a checklist item, not a blocker. Cookie isolation per subdomain works naturally with nuxt-auth-utils.
- Missing operational chapters: per-tenant backup/restore, tenant deletion/offboarding, monitoring, and how migrations behave when tenant #37 of 50 fails mid-deploy (the script continues silently version-skewed). The app.stockapp.com portal login (email → which tenants?) has a TenantUser model but no designed flow.
- Marketing site in the same SPA is awkward — a separate project on the apex domain is cleaner and removes the "landing" branch from the middleware entirely.
- Sessions embed feature flags at login, so plan changes won't take effect until re-login — acceptable, but decide it consciously.

Timeline

8–10 weeks is realistic for the code in phases 1–7 for a solo developer, because the migration is mostly mechanical. It is not realistic through phase 10: provisioning automation, Stripe billing with webhook-driven suspension, the tenant portal, and operational hardening are the long tail. I'd plan 12–16 weeks to a sellable product, or keep 8–10 weeks by cutting billing to "manual invoicing + admin toggles" for the first tenants.

Bottom line: validate-and-proceed. The architecture is right for this codebase and avoids touching production data at all. Fix the Supabase schema-vs-database inconsistency first (with a pooler spike), redesign Phase 5 for serverless, defer the business-rule settings, and add a "migrate existing production customer as tenant #1" chapter — then it's a credible plan.
